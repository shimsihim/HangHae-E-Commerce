## 레이어드 아키텍쳐

### 선정 사유(장점) 
+ 진입 장벽이 낮고 표준화됨
+ 의존성이 단방향으로 명확하여 전체 흐름을 파악하기 쉽고, 쉽게 파악이 가능하다.
+ 각 계층 영역 변경에 대해 다른 계층의 변경을 최소화할 수 있다.

### 단점
+ 엔티티의 경우 데이터 접근계층에 의존하기 떄문에 DB변경 시 비즈니스 레벨까지 변경해야할 수 있다.
+ 서비스 계층에 코드가 집중되며 , 도메인 객체는 데이터 모델로서만 활용될 수 있음.




## 동시성 제어
> 1주차와 동일하게 AOP를 통해 구현 , ReentrantLock사용하였습니다.
> > ReentrantLock 을 관리하는 ConcurrentHashMap의 경우 메모리 누수방지를 위해 만료 및 제거의 구현 또는 카페인캐시를 사용할 수 있다는점을 알아보았습니다.
> > 다만 실제로 사용되는 DB나 분산락이 아닌 인메모리 환경에서의 구현이라는 점에서 추가개발하지 않았습니다.
> 
> > 주문의 경우 도메인의 DB를 변경하며 , 동일 도메인에서도 여러개의 ID가 들어오게 되었습니다.
> > 이로 인하여 쿠폰발급 , 포인트 충전과 같이 특정 Id 요청단위로 Lock을 잡지 못하였으며 해당 UseCase 전체에 락을 잡게되었습니다.
